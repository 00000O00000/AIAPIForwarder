# 全局硬性规则
> 以下规则，除非用户明确说明请违反，否则需要硬性遵守。
1. 全局中文回复：所有输出文本（含回复、代码注释、UI界面文案、日志、提示）**必须使用中文**。
2. 代码阅读规则：分为“通读”和“精读”两种模式。
	- 通读代码规则：新建会话或执行任务前，先快速通读项目以了解大体功能，无需深入细节精读。
	- 精读代码规则：审计或修复 Bug 时必须精读。每次仅读取一个文件（或片段），读取后**立刻**输出分析，完成后再读取下一个。
3. 代码质量回顾：**每轮代码编写/修改完成后**，必须执行**回顾步骤**。重新读取修改后的文件，检查 Bug 和逻辑问题，确保无误。回顾时忽略之前上下文中的代码，以项目实际代码为准，严格遵守`精读代码规则`。
4. 禁止提供 Demo：代码必须完整实现需求，禁止仅写示例。若功能无法实现，必须明确说明并留空相关文件。
5. 禁止运行项目：除非明确允许，否则禁止在当前环境运行项目。环境通常未配置妥当。
6. 遵守 Unix 法则：代码编写与修改必须遵守 Unix 法则（模块化、清晰、组合、分离、简洁、节俭、透明、健壮、最小惊讶、沉默、修复）。高质量代码有助于高效开发。

# 用户偏好规则
> 仅在无明确文档或用户指令，且需要生成时生效。若不需要生成相关内容，或相关内容生成存在明确且无误的定义，请忽略以下规则。
1. 默认前端风格：当项目需要设计前端网页时，如果没有特殊说明，默认按照如下风格进行设计：
- **风格**：**黑白新粗野主义 (Neo-Brutalism) / 线框风格 / 表格式无边距设计**。
- **布局**：全屏填充，允许左右两侧有侧边距，电脑端禁止页面整体滚动（使用内部容器滚动）。
- **元素特征**：
  - 所有容器、按钮、输入框必须有黑边框（1px solid black）。
  - **严格无圆角**（border-radius: 0）。
  - **无外边距设计**（类似于Excel表格的紧凑排列），组件之间共用边框线。
  - 高对比度：纯黑（#000000）与纯白（#FFFFFF），无多余装饰色。
- **响应式布局**：兼容移动端显示；所有UI、提示，均使用自定义的显示，不调用alert等功能。
- **代码编写**：拆分 CSS 文件，避免单文件过大难以维护。

2. 数据库设计检查：存在数据库设计时，需辩证评估。
- **检查数据库设计质量**：若发现用户或文档中的设计存在以下问题，请重新设计
  - 无法完整满足项目需求
  - 不满足第三范式
  - 其他质量低下的情况
- **设计高质量数据库**：设计时需遵守以下核心原则：
  - 需求为先：深入理解业务需求，明确存查改场景。
  - 规范化设计：消除冗余，避免不一致。遵循第三范式（3NF），特定场景允许反规范化以提升性能。
  - 合理命名：表名、字段名直观规范，统一命名规则。
  - 数据完整性：通过约束和外键，维护数据准确一致。
- **告知用户**：设计完成后简要告知用户。若有项目设计文档，直接覆盖更新相关说明。
